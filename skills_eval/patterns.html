Failure-mode patterns & recommended fixes

The same defects appear across many skills. Each pattern below names the failure mode (per the writing-great-skills rubric), lists affected skills, and gives one concrete fix.

Cross-cutting defects

duplicationwidest impact

"When to Activate" / "When to Use" body sections

Affects ≈ 60 of 90 skills.

Nearly every model-invoked skill carries a body section that restates the description's trigger list. The description is the canonical trigger anchor for model invocation; the body section is pure context load with no behavioural payoff.

Fix: Delete the body section in every affected skill. If the body section has a unique trigger the description lacks, lift it into the description and then delete the section.

Examples: config-gc, browser-qa, exa-search, documentation-lookup, frontend-a11y, react-patterns, error-handling, dotnet-patterns, market-research, frontend-slides, motion-foundations.

sprawl

400+ line bodies with zero progressive disclosure

8 skills exceed 400 lines without disclosing branch-specific reference.

Large pattern/reference skills inline every code example, every appendix, and every alternate-path block. Pruning is impossible because nothing has been split out. The agent pays full context load on every invocation even when only one branch is reached.

Fix: For each skill, identify branch-specific sections (a code language, a platform, an advanced sub-topic) and disclose them to sibling files reached by a named context pointer. Target SKILL.md under 150 lines.

no-op / weak leading word

Descriptions that enumerate content rather than fire on triggers

≈ 12 skills.

The description lists what the skill covers ("hooks, server/client seams, Suspense, forms…") instead of stating when the agent should reach for it. The result is unreliable invocation — the model has no leading word to hang the decision on.

Fix: Front-load each description with a pretrained leading concept and a trigger clause. Rewrite "Use this skill to…" openers; they bury the anchor.

  • git-workflow, frontend-patterns, flutter-dart-code-review, dart-flutter-patterns, dotnet-patterns
  • Homelab cluster: homelab-network-setup, homelab-pihole-dns, homelab-vlan-segmentation, homelab-wireguard-vpn
  • browser-qa, resolving-merge-conflicts, skill-stocktake all open with "Use when…" instead of leading word first
premature completion

Workflow steps with no checkable completion criterion — only where the domain is closed

Narrower than initially scoped. The rubric says "where it matters" and explicitly allows irreducibly fuzzy criteria; the defence for those is splitting the sequence, not forcing a fake bar.

A hard completion criterion fits when the set of things to check is knowable up front (every CSS file, every named dimension, every signal field). On open-ended work — Socratic dialogue, exploratory research, design diagnostics — a premature criterion creates a false ceiling: the agent stops at the bar it was given and misses what the work itself would have surfaced.

Add a hard bar (closed domain):

  • brand-voice: "Every VOICE PROFILE dimension has ≥ 2 source examples confirming it" — dimensions are fixed in the schema.
  • lead-intelligence Stage 1: "≥ N scored prospects with all six signal fields populated" — fields are fixed.

Leave open (irreducibly fuzzy by design):

  • grill-with-docs — Socratic interrogation. New terms surface because of the grilling; a "canonicalize every term" bar would freeze the agent on the user's first message.
  • deep-research Step 5 — synthesis is emergent. Earlier steps (define sub-questions, search depth) can carry bars; the synthesis step cannot.
  • product-lens — diagnostic dialogue. Done is judgement, not a checklist.
  • handoff — handoff shape is context-dependent; naming required sections would constrain useful variation.

For the irreducibly fuzzy cases: if the agent is observed rushing, hide post-completion steps by splitting the skill so attention can't slip ahead. Do not invent a criterion to fill the gap — that's worse than none.

What the top-quartile skills do differently

Pretrained leading words

The top skills anchor entire regions of behaviour on a single pretrained concept: caveman, tight, tracer bullet, PICO, garbage collection, throwaway, baseline / increment / trend. The body uses them repeatedly so the model can hang its execution on the prior.

Disclosure that mirrors branches

writing-great-skills, prototype, codebase-design, tdd, orch-add-feature, and remotion-video-creation all push branch-specific reference behind named context pointers. The body of each skill stays under 150 lines.

Checkable completion criteria

The top workflow skills end each phase on a binary observable: diagnosing-bugs (loop goes red or not), align-vocabulary ("every drifting term substituted or explicitly retained"), network-config-validation (5-pass ordered check), orch-fix-defect (reproduce as failing test, then green).

Front-loaded descriptions

The strongest descriptions lead with the skill concept itself rather than "Use this skill to…". They list distinct branches without synonym inflation and contain no body-duplicating exposition.

Suggested cleanup pass — priority order

  1. Sweep deletion of "When to Activate" body sections across the ≈60 affected skills. Highest-impact single change; restores description as single source of truth for invocation.
  2. Progressive disclosure on the 8 sprawl skills (lines 400+). Cuts steady-state context load substantially.
  3. Description rewrite pass: front-load leading word, delete synonym lists, strip "Use this skill to…" openers.
  4. Add completion criteria to workflow skills missing them.
  5. Targeted fixes: retire/scope motion-ui, remove out-of-scope sections from coding-standards, align scientific-thinking name/slug pairs, delete orch-* "Actor · action · target" lines, set aside to user-invoked.
